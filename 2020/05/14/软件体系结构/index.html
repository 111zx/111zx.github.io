1. 数据流体系结构

概述

  • 系统模块之间传递的数据：数据流
  • 无数据时不工作，数据到达时即被激活

特点

  • 数据的可用性决定着处理计算单元是否进行
  • 系统结构由数据在各处理之间的有序移动决定
  • 在纯数据流系统中，处理之间除了数据交换外没有任何其他的交互

基本构件

  • 数据处理单元
    • 输入端口：读取数据，在计算单元内计算，传给输出端口
    • 输出端口：输出数据
  • 连接件：数据流
    • 连接两个数据处理单元，通常是异步，有缓冲
    • 由reader和writer
    • 把一个数据单元的输出接口和下一个的输入端口连接
  • -数据流动模式
    • 自由流动（非常容易造成死锁，一般不是刻意用数据流体系结构风格完成的）
    • 近似线性流动（这种流动是正确的数据流风格选择）
    • 在限定内循环数据流

批处理风格

概述

  • 每个步骤完成计算之后，把所有完整的数据拷贝到一个载体上，传给下一个步骤，循环。
  • 每一步必须全部完成才可以进行下一步
  • 每一步是独立的应用程序
  • 数据必须完整传输

基本构件

  • 独立的应用程序

连接件

  • 某种类型的媒介

拓扑关系

  • 连接件所定义的数据流图，表达拓扑关系

应用实例

  • 基于Eclipse的代码重复检测工具
    • 代码分析：两个源代码
    • 程序转换：抽象语法树
    • 相似代码检测：P—String文件
    • 相似代码显示：结果文件
    • 度量分析

管道-过滤器风格

概述

  • 不需要把整体的数据传输。例如：自来水处理，边处理边传输
  • 数据原点产生数据后在不同的过滤器过滤之后传给下一个步骤

应用场景

  • 数据源源不断产生，系统需要对这些数据进行若干个步骤的处理（分析，计算，转换等）

解决方案

  1. 把系统分解为几个处理步骤，每个处理步骤之间由数据流链接，一个步骤的输出是另一个步骤的输入
  2. 每个步骤是一个过滤器
  3. 步骤之间数据的传输是管道

基本构件

  • 过滤器
    • 目标：将接收到的数据流转换成目标数据流
    • 5种转变类型
      • 计算和增加信息
      • 浓缩和删减信息
      • 改变数据的表现方式
      • 分解数据流
      • 合并数据流
    • 过滤器是独立的构件
      • 上下文无关
      • 不保留任何数据的状态
      • 对其他过滤器无任何了解

连接件

  • 管道
    • 目标：运输数据流
    • 单向流
    • 可能有缓冲区
    • 管道之间的数据格式不固定（有没有可能带来什么问题？）

拓扑关系

  • 管道和基本构件形成的数据流图

特点

  • 递增的读取和消费数据流，数据有便处理，处理完数据即传输，不需要等待
  • 隐蔽性，高内聚，低耦合
  • 支持软件复用
  • 系统的维护和增强系统性能简单
  • 支持并行
  • 允许对吞吐量，死锁等属性的分析
  • 缺点
    • 不适合处理交互的应用
    • 系统性能不高，并增加了编写过滤器的复杂性
      • 绝大部分时间消耗在了格式转换上
      • 不适用大量共享数据的应用设置

批处理和管道-过滤器对比

共同点

  • 都是分解任务为固定计算单元
  • 每个计算单元之间的交互都只有数据的传输

不同点

2. 调用返回体系结构风格

概述

  • 主程序/子程序风格
    • 单线程控制，划分为若干处理步骤
    • 功能模块：把步骤聚集再模块中
  • 抽象数据类型
    • 数据和操作绑定在一起，隐藏实现和其他秘密
  • 面向对象
    • 对象活动与不同的线程和进程
    • 方法（动态绑定），多态（子类），重用（继承）

适合的场景

  • 有层次化，可以分解的应用系统

上下文

  • 主流编程语言都支撑风格

解决方案

  • 组件内部的数据和功能都是可见的

  • 过程调用

  • 单线程

模块分解

  • 功能实现在模块内部，外部不知道内部的变化

封装/信息隐藏

  • 自己负责状态的一致性
  • 通过接口访问
  • 模板
  • 以一种可以控制的状态允许修改

数据抽象/面向对象

  • 去中心化

面向对象体系结构的元素

  • 封装
  • 交互
  • 多态
  • 继承：共享的功能保持唯一的接口
  • 复用和维护：利用封装和聚合提高生产力

面向对象的问题

  • 对象海洋需要额外的结构来容纳
  • 单个接口有限且笨拙
如何管理大的对象数据集
  • 等级制度

层次风格

案例

  • 操作系统
    • 应用平台，应用，操作系统
  • 计算机网络
    • 计算机网络7层结构

问题

  • 层次划分明显

上下文

  • 每一类具有相同的结构放在一起

解决问题

  • 单线程
  • 过程调用

特点

  • 每层为上一层提供服务，只能看到和自己邻接的层
    • 必要时可以允许一层的跃层
  • 大的问题分解为若干小问题，逐步解决，隐藏复杂度
  • 修改一层，最多影响2层，而通常之影响上层。接口稳固，则谁都不影响
  • 上层必须知道下层的身份，不能调整层次之间的顺序
  • 层层相调，影响性能

C/S架构

两层C/S（胖客户机）

组成部分
  • 服务器主要存储数据库
  • 客户端负责业务逻辑
  • 网络
缺点
  • 客户端配置要求高，程序复杂
  • 客户端可以直接访问数据库，不安全
  • 用户界面风格不一，不利于推广
  • 软件维护和升级困难，每个客户端都要维护升级

三层C/S（瘦客户机）

组成部分
  • 服务器主要存储数据库：数据层
  • 客户端负责不再业务逻辑，只是一个展示的作用：展示层
  • 应用服务器负责业务逻辑：功能层

B/S

  • 客户端只能从客户端拉数据而不能向服务器推数据。
  • 安全性难控制（SQL注入）

3. 数据中心体系结构风格

概述

  • 注册表是一种数据中心体系结构风格
  • 在注册表未问世之前，系统所有的硬件和配置信息都被保存在各自的配置文件中，散落在系统的各个角落，有了注册表的思想，就将配置文件集中起来存储，形成了共享仓库。
  • 剪贴板也是一个数据为中心的系统，可以共享数据

仓库体系结构风格

概述

  • 仓库是存储和维护数据的中心场所
  • 基本组件
    • 中心数据结构
    • 在中心数据结构上操作的独立构建

连接件

  • 仓库和独立构建之间的交互

交互机制

  1. 数据库方式：输入流中的事务类型触发需要执行的过程
  2. 黑板结构：中心数据结构的当前状态触发并选择需要执行的过程

黑板体系风格

适用场景

  • 一个大问题被分解为若干个子问题
  • 每个子问题的解决需要不同的问题表达方式和求解模型，分别涉及求解程序。

基本构件

  • 黑板
  • 控制器
  • 知识源

知识源

  • 知识源是描述某个独立领域问题的知识及其处理方法的知识库，其分别存放且相互独立的，他们通过黑板进行通讯，合作求出问题的解，通常知识源具有条件动作的形式。当条件满足时，知识源被触发，其动作部分增加或修改黑板上得内容。
    • 待解决的问题被分为若干个子问题，每个子问题由一个独立的知识源加以理解
    • 知识源包含独立的领域知识
    • 知识源执行计算后会跟新黑板里的数据状态
    • 多个知识源 之间只能通过黑板交换知识

控制器

  • 时刻监视黑板状态变化
  • 对黑板上信息的当前状态进行判断和评价
  • 当黑板的状态满足了黑板源的执行条件时，该知识源被控制器触发并进行计算，然后将结果更新到黑板上
  • 这种更新又导致其他知识源参与计算并更新黑板，直到找到问题解为止

应用场景

  • 自然语言处理
  • 语音处理
  • 模式识别
  • 图像处理

5. 事件系统体系结构风格

事件

  • 能够激活对象功能的动作，当发生这种动作后将给所涉及对象发送一个消息，对象便可执行相应的功能。

事件系统

主要特点 - 隐式调用

  • 事件的触发者并不知道那些构件会被这些事件影响，相互保持独立
  • 不能假定构建的处理顺序，甚至不知道那些过程被调用
  • 各个构件之间彼此无连接关系，各自独立存在，通过对事件的发布和注册实现关联

基本结构和工作原理

  • 事件源发布时间，事件管理器通知时间给事件处理器处理
  • 特点
    • 分离的交互：时间发布者并不会意识到事件订阅者的存在
    • 一对多通信：采用发布/订阅消息传递，一个特定时间可以影响多个订阅者
    • 基于事件的触发器：由事件出发过程调用
    • 异步：支持异步操作

基本构件

  • 事件源 ：发布事件
  • 事件处理器：向特定的事件进行注册
  • 事件管理器

连接机制

  • 一个事件也可能会触发其他事件，形成事件连

应用实例

调试器中的断点机制

  • 基本构件
    • 事件源：调试器
    • 事件处理器：编辑器与变量监视器
    • 事件管理器：集成开发环境
  • 处理过程
    • 编辑器与变量监视器向调试器注册，接受断点事件
    • 遇到断电，调试器发布事件，从而触发编辑器与变量检测器
    • 编辑器将源代码滚动到断点处
    • 变量检测器则跟新当前变量值并显示出来

调度机制

无独立调度模块的事件系统

  • 观察者/被观察者
  • 每一个模块都允许其他模块向自己所能发送的某些消息表明兴趣
  • 当某一模块发出某一事件时，他自动将这些事件发布给那些曾经向自己注册过此事件的模块

有独立调度模块的事件系统

  • 事件派遣模块
    • 负责接收到来的事件并派遣他们到其他模块
  • 派遣方式
    • 全广播式
      • 派遣模块将时间广播到所有的模块，但只有感兴趣的模块才取出事件并触发自身的行为
    • 选择广播式
      • 派遣模块将时间送到那些已经注册的模块中
选择广播式的两种策略：基于事件被执行的方式
点对点模式：基于消息队列（一对一）
  • 其他应用向消息队列发布事件，队列管理器会在消费者将他注册过的事件取回后删除事件。
发布订阅模式（一对多）
  • 其他应用向消息队列发布事件，队列管理器会在消费者将他注册过的事件取回搁浅到事件过期后再将事件删除

1. 可用性

定义

  • 当用户使用系统时，系统可用的概率
  • 提前确定的停机维护不计入

场景

刺激源

  • 故障的迹象（来自内部或外部）

刺激

  • 系统出错
  • 系统崩溃
  • 给出结果不准时
  • 给出错误结果

制品

  • 网络传输 or 计算 or 存储

环境

  • 正常状态 or 亚健康状态

响应

  • 记录日志，回传给厂家
  • 通知管理员或其它系统
  • 关闭系统，系统在维修期间不可用

响应衡量 指标

  • 故障时间百分比，修复故障所需时间，平均无故障时间

提升的策略

方向1 故障检测

  • Ping/echo
  • Heartbeat
    • 定期发出心跳消息，未收到则说明出现异常
  • Exceptions
    • java异常处理

方向2 故障恢复

  • 投票
    • 多个组件完成同一任务，少数服从多数
  • 主动冗余
    • 备胎，A出错用B，A和B每时每刻都是保持一致
  • 被动冗余
    • A将状态告诉B，A出错，B可以换A
  • 内测
    • 游戏内测检查BUG
  • 检查点/回滚
    • 定期保存

方向1 故障避免

  • 服务下线
    • 知道要被攻击，主动下线
  • 事务
    • 多个操作必须全部完成才结束
  • 进程监控
    • 任务管理器

2. 可修改性

关注点

  • 修改的成本
  • 系统的哪些部分被修改
  • 修改发生的时间
  • 修改由谁来进行

刺激源

  • 谁进行的修改（开发者/管理员/用户）

刺激

  • 要进行的具体修改

制品

  • 修改系统的功能 or UI or 交互的其它系统

环境

  • 在什么时间进行修改？设计期间 or 开发期间 or 运行期间

  • 修改的时间越迟，越不利

响应

  • 操作人员要理解如何修改，进行修改操作，测试，部署

响应衡量指标

  • 时间，成本

提升可修改性的策略

  • 目标：降低修改的时间和成本

方向1 限制修改的范围

  • 模块高内聚，低耦合

  • 考虑到可能发生的修改

  • 让模块通用

  • 隐藏信息

  • 维持接口不变

  • 限制通信路径

    • Facade模式

  • 使用中介

    • 数据中介
    • 服务中介：设计模式的bridge，工厂方法模式

提升可修改性的策略

  • 配置文件
  • 发布订阅模式
  • 多态

3. 性能

关注点

  • 系统响应时间的速度

  • 和时间的数量和到达模式有关

事件的来源

  • 用户的请求，本系统内部，本系统外部

六个要素

  • 刺激源：系统内部或外部
  • 刺激：事件到来
  • 制品：系统所提供的服务
  • 环境：系统处于不同的模式（正常/紧急/超载）
  • 响应：系统处理到来的时间，可能会导致状态的变化
  • 响应衡量指标：
    • 处理事件所花的时间
    • 单位时间内处理时间的数目
    • 出库的错误率/丢失率

提升的策略

目标

  • 在限定时间内响应事件
  • 获取资源 + 使用资源

方向1 资源的需求

  • 数据量不变，提高计算效率
    • 算法效率
    • 减少处理事件时对资源的占用
  • 减少总数据量
    • 控制事件到达速率
    • 之抽取一部分请求来处理
  • 限制执行时间
    • 在规定时间内得到近似解
  • 限制待处理事件队列长度
    • 直接放弃一部分事件

方向2 资源的管理

  • 利用并发机制
  • 增加可用资源
  • 动态优先级

方向3 资源的仲裁

4. 安全性

关注点

  • 在保证合法用户使用系统的前提下，抵抗对系统的冲击

六大要素

  • 刺激源：攻击可能由人或其它系统发起
  • 刺激：对系统的攻击
  • 制品：系统所提供的服务或者系统中的数据
  • 环境：系统可能处于不同的情况下
  • 响应：合法用户正常使用，拒绝非法用户使用
  • 响应衡量指标：发起攻击的难度；从攻击中回复的难度

提升的策略

方向1 抵抗攻击

  • 用户验证
  • 用户授权
  • 加密传输：https
  • 维持数据的完整性：MD5码校验
  • 减少暴露信息
  • 限制访问

方向2 检测攻击

  • 软件和人结合
    • 安全专家
    • 360

5. 可测试性

关注点（软件测试是为了发现错误）

  • 让软件的bug容易被测试出来
  • 验证软件产品与它的需求规格是否匹配
  • 使用最小的成本和工作量来验证软件的质量

六要素

  • 刺激源：各种角色都有可能发起（开发者，单元测试人员，集成测试人员，系统管理人员，用户）
  • 刺激：系统开发到了里程碑
  • 制品：一个设计，一段代码，整个系统
  • 环境：系统可能处于设计/开发/部署/正常运行阶段
  • 响应：理想的响应式可以进行测试，并且可以观察到测试结果；当测试结果无法被观察到时，测试难度很大
  • 响应衡量指标：白盒测试中的覆盖率；未来继续发现BUG的概率

提升可测试性的策略

目标

  • 让测试更加轻松

方向1 黑盒测试

总体思路：提供输入 + 捕获输出

  • 记录/回放：自动化/半自动化测试
  • 把接口和实现分离开：不同的排序算法，相同的接口
  • 提供专用的测试路径

方向2 白盒测试

  • 内部监控
    • IDE提供的断点等调试工具

第三章 软件体系结构评估

软件体系结构评估概述

1. 为什么要进行软件体系结构评估？

  • 体系结构可能存在潜在风险，有可能影响到以后的开发成本和需求满足。

2. 定义

  • 架构评估是开发生命周期的活动,一些利益相关者(项目干系人)分析软件架构在一个正式的或非正式的(正式或非正式)过程使用评估技术,如场景(场景)

3. 好处

  • 对现有体系结构问题的早期检测，检测到问题的生命周期越早，修复问题的成本就越低。

4. 怎么样进行评估？

  • 利用各种评估方法
  • Scenario-based Architecture Analysis Method (SAAM)
    • 这可能是第一个文档化的软件架构分析方法，最初是为了分析架构的可修改性而开发的。但是，它对于分析架构的任何非功能方面都是有用的。它基于使用利益相关者生成的场景来评估架构

ATAM

1. 概述

  • ATAM是一种帮助涉众提出正确问题以发现潜在问题架构决策的方法

2. 目标

  • ATAM的目的不是提供精确的分析，其目的是发现架构决策所产生的风险

3. 参与人员

  • 评估小组：
    • 每个评估团队由一名领导和至少三名其他团队成员组成。
    • ATAM团队成员必须是有经验的架构师。
    • ATAM的领导必须有优秀的沟通和引导技巧。
  • 项目决策者
    • 能够控制程序开发过程和授权变更的人员
    • 通常包含项目经理,为开发买单的用户,和体系结构开发人员
  • 架构涉众
    • 涉众是与要开发的系统相关的一切人和事
    • 涉众不等于用户

4. 具体步骤

0 阶段（在技术评估之前，是准备工作）
  • 主要工作：客户和评估团队协商评估的方法和系统信息，确定评估方法，最后部署评估工作
1 阶段

  • 概述：是以体系结构为中心对架构的详细信息进行自顶向下分析

  • 参与人员：主要是技术相关人员

  • 具体步骤

    1. 评估小组介绍ATAM方法
    2. 客户描述需求（功能性和非功能性需求）
    3. 设计人员介绍架构的设计
    4. 确定架构的评估方法
    5. 通过构建一个效用树来识别、优化最重要的质量属性
    6. 评估团队从质量属性的角度探索架构方法，以识别和记录风险和non-risks,敏感点和权衡点
      • 敏感点：一个或多个组件(和/或组件关系)的属性，对于实现特定质量属性响应至关重要
      • 权衡点：影响多个质量属性的属性，并且是多个属性的敏感点
      • 风险：是一个潜在的有问题的架构决
      • non-risks：是良好的架构决策，在分析之后被认为是安全的。

  • 效用树

    • 定义：自顶向下对质量属性进行分析的工具，用来刻画需求
    • 叶子节点：场景描述
    • 根节点的子节点：质量属性
    • 中间节点（倒数第二层）：优先级

  • 产出：

    ➢架构方法

    ➢效用树

    ➢场景

    ➢风险和非风险

    ➢敏感性和权衡

2 阶段

  • 参与人员：各种干系人

  • 具体步骤

    1. 干系人头脑风暴产生场景描述
    2. 利用产生的场景对1阶段产生的成果再一次进行分析
    3. 概括所有步骤并完成输出

  • 产出：

    ➢架构方法

    ➢效用树

    ➢场景

    ➢风险和非风险

    ➢敏感性和权衡

    ➢风险主题

3 阶段

  • 参与人员：技术人员

  • 概述：主要包括为客户制作最终报告，并对评估的质量和ATAM材料进行反思

  • 最终产出：

    ➢执行概要

    ➢ATAM的描述

    ➢需求和架构的描述

    ➢列出阶段1和阶段2的场景和效用树

    ➢阶段1和阶段2分析:架构方法,决策,风险,敏感性,权衡,non-risks

    ➢风险主题

    ➢下一步

第四章 应用案例——软件体系结构风格

课前问题

  • 在结束体系结构评估之后，软件体系结构的设计过程已经基本结束

KWIC案例

需求获取

  • 功能性：输入 -> 循环位移 -> 排序输出
  • 非功能性：可修改，系统复用性

1. 主程序 - 子程序风格

主要思想

  • 对系统进行功能分解，是最自然的想法,也是“面向过程的编程” 的主要思路
  • 分为四个基本功能：输入、移位、排序、输出
  • 主程序按次序调用这四个模块
  • 通过共享的数据存储和无约束的读-写协议在模块之间进行数据交换

优缺点

  • 优点
    • 模块之间的数据共享
    • 不同的计算功能被隔离在不同的模块中
  • 缺点
    • 对数据存储格式的变化将会影响几乎所有的模块
    • 对处理流程的改变与系统功能的增强也很难适应，依赖于控制模块内部的调用次序
    • 这种分解也难以支持有效的复用

2. 面向对象风格

主要思想

  • 数据不再被构件直接共享，而是被封装在了Object中
  • 每个对象提供了一个接口，允许其他对象通过该接口调用对该对象内封装的数据的操作
  • 主要对象
    • Input object：负责从输入文件中读取数据并将其存储在LineStorage对象
    • LineStorage object：存储和处理字符、单词、行
    • CircularShifter object：负责对LineStorage对象中存储的数据进行循环移位
    • Alphabetizer object： 负责对循环移位后得到的数据进行排序
    • Output object：负责打印输出排序后的数据)
    • Master control object ：负责控制其他各对象中方法的调用次序

优缺点

  • 优点
    • 某一构件的算法与数据结构的修改不会影响其他构件
    • 构件之间依赖性降低，提高了复用度
  • 缺点
    • 不是特别适合功能的扩展
    • 为了增加新功能，要么修改已有的模块，要么就加入新的模块

3. 管道过滤器风格

主要思想

  • 四个过滤器：输入、移位、排序、输出
    • “输入”过滤器：从数据源读取输入文件，解析格式，将行写入输出管道
    • “循环移位”过滤器
    • “排序”过滤器
    • “输出”过滤器
  • 每个过滤器处理数据，然后将结果送至下一个过滤器
  • 控制机制是分布式的：只要有数据传入，过滤器即开始工作
  • 过滤器之间的数据共享被严格限制在管道传输

优缺点

  • 优点
    • 过程流非常直观
    • 支持复用
    • 容易修改
    • 过滤器的功能相互隔离
    • 新功能容易加入
  • 缺点
    • 无法支持交互式系统，局限性较大
    • 空间复杂性高
