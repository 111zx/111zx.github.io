概述

设计模式的应用场景

  • 设计模式是对软件设计中普遍存在的问题，提出的解决方案。

  • 设计模式可以将软件规范化，易于理解，增强复用性，维护性

  • 面向对象=>功能模块（设计模式+算法）=>框架（多种设计模式）=>架构（服务器集群）

设计模式七大原则

设计模式的目的

  • 代码重用性
  • 可读性
  • 可扩展性
  • 可靠性
  • 高内聚（模块内凝聚）低耦合（不同模块之间依赖性小）

单一职责原则

  • 一个类应该只负责一项职责（不是只有一個方法，是一項職責（訂單的類不要去操作員工的職責））；如果一个类负责两个不同职责，职责1改变，导致职责二执行错误。

  • 职责也可以用类里面的方法分解。（类的结构简单的情况下）

  • 特点

    • 降低类的复杂性
    • 提高可读性，可维护性
    • 降低变更引起的可能性
    • 只有类的结构很简单，方法足够少时，才能根据方法进行单一职责分解。

  • 用類來化解if else分支

接口隔离原則

  • 客户端不应该依赖它不需要的接口，即一个类对另一个类的依赖应该建立在最小的接口上
  • 拆分接口，各取所需

依赖倒转原则

关注点

  • 高层模块不应该依赖底层模块，二者都应该依赖其抽象
  • 抽象不应该依赖细节，细节应该依赖抽象
  • 依赖倒转的中心思想是面向接口编程
  • 相对于细节的多变性，抽象的东西要稳定的多。一抽象为基础搭建的架构比以戏界为基础的架构要稳得多。在java中，抽象指的是接口或者抽象类，细节就是具体的实现类。
  • 在接口不抽象类里面不要有具体的操作

总结

  1. 一层模块尽量都要有抽象类或接口，或者两者都有，程序稳定性更好
  2. 在声明对象的时候最好使用接口或者抽象类，这样在变量引用和实际对象之间会有一个缓冲层，有利于程序扩展和优化
    • 在类继承多个接口时，使用接口定义引用变量类型可以清楚了解这个变量的含义
    • 想换一种实现类的构造器来实现功能时，只需要修改构造器的名字
    • 如果类内实现了接口没有的方法，那么不能使用接口定义变量
  3. 继承时遵循里氏替换原则

里氏替换原则

概述

  • 继承性的劣势
    • 对象间的耦合性：如果父类需要改变，那么子类的方法可能需要大面积修改
  • 原则基本介绍
    • 尽量不要去重写父类的方法
    • 如果有必须的功能要求，可以通过聚合，组合，依赖来解决，降低耦合性。可以将基类的方法优化到最简，可以降低耦合性。

开闭原则

基本介绍

  • 最基础，最重要的设计原则
  • 对提供方开放，对使用方关闭
  • 模块和函数应该对扩展开放，对修改关闭。用抽象构建框架，用实现扩展细节。
  • 当软件需要变化时，尽量通过扩展软件实体的行为来实现变化，而不是通过修改已有的代码来实现变化
  • 使用设计模式的目的就是遵循开闭原则

迪米特法则（最少知道原则）

基本介绍

  • 在别人的类里不要有自己类的业务逻辑，只提供完成业务逻辑的方法的接口。

  • 一个对象应该对其它对象保持最少的了解

  • 类与类的关系越密切，耦合度越大

  • 一个类对子依赖的类知道的越少越好。对于被以来的类不管多麽复杂，都精良讲逻辑封装在类的内部。对外除了提供的public方法，不对外泄露任何信息。

  • 耦合的方式

    • 依赖
    • 关联
    • 组合
    • 聚合

  • 一个类在另一个类中作为成员变量，方法参数，方法返回值中的类为直接朋友

  • 出现在局部变量中的类不是直接朋友，例如在方法中new出来的类

  • 降低耦合，并不是消除耦合

合成复用原则

  • 尽量使用聚合，而不是集成
    • 如果B类需要使用A类的方法，继承的话耦合性会很高
    • 可以在B类里传入一个A类的参数，调用A类的方法
    • 可以在B类里定义一个A类的对象，然后使用A类的方法，这种方式是组合
    • 聚合：在B类声明一个A类参数，然后声明一个set方法去实例化A类，然后就可以使用。

总结

  • 将需要变化的和不需要变化的部分区分开来设计
  • 针对接口设计，而不是针对实现编程
  • 所有的目的是为了松耦合

Git

概述

git-版本控制器

  • 改代码思路错误之后需要回滚

  • 找出谁修改的代码导致的BUG

集中化的版本控制（svn）

  • 服务器保存所有文件的修订版本
  • 每个人都可以看到其他人的代码，管理员也可以掌控每个开发者的权限
  • 存储的时每个相邻版本之间的差异，需要硬盘的空间会相对的小一点，回滚的速度会很慢
  • 优点：
    • 代码存放在单一的服务器上，便于项目管理
  • 缺点：
    • 服务器宕机：员工写的代码得不到保障
    • 服务器故障：整个项目历史记录都会丢失

分布式版本控制（git）

  • 客户端并不只提取最新版本的文件快照，而是把代码仓库完整镜像下来
  • 客户端存的是索引，而已磁盘空间很小，每个客户端都可以放下整个项目的历史记录
  • git每次存储的都是项目的完整快照，需要按的硬盘空间会相对大一点（Git对代码进行了极致的压缩，最终的实际空间比svn多不了多少，而且git回滚速度快）

JVM的架构模型

概述

基于栈特点

  1. 简单，资源受限
  2. 避开了寄存器的分配难题
  3. 指令流中的指令大部分是零地址指令，其过程依赖于操作栈
  4. 不需要硬件支持，可移植性好

基于寄存器架构的特点

  1. 典型的是x86的二进制指令集
  2. 指令集架构完全依赖于硬件，可移植性查
  3. 性能优秀更高效
  4. 花费更少的指令去完成一项操作
  5. 大部分情况下，多以一地址，二地址和三地址指令执行。

总结

  • 栈：跨平台性，指令集小，指令多；执行性能比寄存器差
  • Java的指令都是根据栈来设计的

虚拟机栈

1. 概述

  • 出现背景：由于跨平台的设计，Java指令都是根据栈来设计的。不同平台CPU架构不同，所以不能设计基于寄存器的。
  • 优点：跨平台，指令集小，编译器容易实现
  • 缺点：性能下降，实现同样功能的指令复杂多样
  • 为什么人们提到jvm内存结构只有堆和栈
    • 首先是错误的，不全面的
    • 说明栈和堆比较重要
    • 栈解决程序的运行问题，即程序如何运行，或者说如何处理数据
    • 堆解决的是数据存储的问题，即数据怎么放，放在哪
  • 通常堆空间是运行时空间比较大的一部分
  • 每一个线程都会有一个虚拟机栈，内部保存的是一个个的栈帧
    • 每个栈帧对应一个java方法
  • 是线程私有的
  • 生命周期随线程的消失而消失，生成而产生

2. 作用

  • 主管程序的运行，保存方法的局部变量（8种基本数据类型，对象的引用地址），部分结果，并参与方法的调用和返回
    • 局部变量 vs 成员变量
    • 基本数据变量 vs 引用数据变量

3. 栈的优点

  • 仅次于PC寄存器的速度
  • 只有进栈和出战两个操作
  • 不存在垃圾回收问题，但是有内存溢出

虚拟机栈的内容

1. 栈内存什么？

  • 每个线程都有自己的栈，栈中的数据以栈帧的格式存在。
  • 每个线程上执行的每个方法都各自对应一个栈帧。
  • 栈帧是一个内存区块，是一个数据集，维系着方法执行过程中的各种数据信息。

2. 栈运行原理

  • JVM对栈的操作只有两个，压栈和出栈。

  • 在一条活动线程中，只会有一个活动的栈帧。

    • 当前栈帧：当前执行方法的栈帧是有效的。
    • 当前方法：和当前栈桢对应的方法。
    • 当前类：定义当前方法的类。

  • 执行引擎运行的所有字节码指令只对当前栈帧有用。

  • 不同线程中所包含的栈帧是不允许存在相互引用的，即不可能在一个栈帧中引用另外一个线程中的栈帧。

  • 如果当前犯法调用了其他方法，在方法返回之际，当前栈帧会传回此方法的执行结果给钱一个栈帧，接着，虚拟机会丢弃当前栈帧，使得前一个栈帧重新成为当前线程。

  • Java方法有两种返回函数的方式，一种是正常的函数返回，使用return指令；另一种是抛出异常。不管是用那种方式，都会导致栈帧被弹出。

    • 如果方法中出现异常，抛出异常给上一个方法，并结束。
    • 每个方法结束是以return；不加只是省略而已，并不是不存在。

3. 栈帧的内部结构

  • 局部变量表
  • 操作数栈（表达式栈）
  • 动态链接（指向运行时常量池的方法引用）
  • 方法返回地址（正常退出或异常退出）
  • 一些附加信息

  • 程序：完成特定任务，某种语言编写的一组指令的集合。是一个静态的代码

  • 进程：程序的一次运行状态。是一个动态的过程。有它自身的产生，存在和消亡的过程

  • 线程：进程可以划分为线程，是一个程序内部的一条执行路径。可以并行。

    • 线程拥有独立的运行栈（虚拟机栈）和程序计数器。
    • 方法区和堆是进程的，线程可以共享。
    • 多线程运行共享同一个进程的内存单元，所以会带来安全隐患。

  • 单核CPU和多核CPU的区别

    • 单核：假的多线程，时间片轮流处理
    • 多核：多核处理，真多线程
    • 一个java.exe。至少由三个线程组成：main()主线程，gc垃圾回收线程，异常处理线程。

  • 并发与并行

    • 并行：多个CPU同时执行多个任务
    • 并发：时间片轮流处理

  • 多线程的优点(以单核DPU为例)

    • 提高应用程序的响应。对图形界面更有意义，可增强用户体验。
    • 提高CPU利用率
    • 改善程序结构：把复杂的进程分成多个线程独立运行，立于理解

1. Java内存模型

1-1. 概述

  • 每一个线程都有一个自己的工作内存，里面存的是从主内存取来的共享变量副本

1-2. JMM数据原子操作

  • read():从主存中读取数据
  • load():将主存读取到的数据写入工作内存
  • use():从工作内存读取数据来计算
  • assign():将计算好得知重新复制到工作内存里
  • store():将工作内存数据写入主存
  • write():将store过去的变量值赋值给主内存中的变量
  • lock()：将主内存变量加锁，表示为线程独占状态
  • unlock()：将主内存变量解锁，解锁后其他线程可以锁定该变量

1-3. 总线枷锁（性能低，被淘汰）

  • 实现了数据同步，但是并行改变为串行，性能变低

1-4. MESI缓存一致性协议

  • CPU的总线监听机制，如果发现总线上的变量变动后，会将自己线程内的相应变量失效掉
  • 当数据失效后，CPU只能从主内存中read，load变量。

2. volatitle 内存可见性实现原理

  • volatitle是C语言实现的

  • 通过一个lock前缀指令（和lock原子操作不同，会触发cpu发生lock原子操作），会锁定这块内存区域的缓存并写回内存

  • lock前缀指令：

    • 会开启CPU监听机制
    • 会将处理器缓存行的数据立即写回到主内存，以保证线程的共享数据的一致性
    • 这个协会内存的操作会引起其他线程共享变量副本数据的无效

并发编程的三大特性

  • 可见性
  • 原子性
    • volatile不能保证变量的原子性，需要使用sychronized同步机制
      • 在写写操作时会出现错误
  • 有序性

原子变量与CAS算法

  • volatile保证内存可见性，可以通过原子变量解决，CAS算法(比sychronized效率高)能保证数据的原子性。
  • AtomicInterger关键字声明int变量

1. 数据流体系结构

概述

  • 系统模块之间传递的数据：数据流
  • 无数据时不工作，数据到达时即被激活

特点

  • 数据的可用性决定着处理计算单元是否进行
  • 系统结构由数据在各处理之间的有序移动决定
  • 在纯数据流系统中，处理之间除了数据交换外没有任何其他的交互

基本构件

  • 数据处理单元
    • 输入端口：读取数据，在计算单元内计算，传给输出端口
    • 输出端口：输出数据
  • 连接件：数据流
    • 连接两个数据处理单元，通常是异步，有缓冲
    • 由reader和writer
    • 把一个数据单元的输出接口和下一个的输入端口连接
  • -数据流动模式
    • 自由流动（非常容易造成死锁，一般不是刻意用数据流体系结构风格完成的）
    • 近似线性流动（这种流动是正确的数据流风格选择）
    • 在限定内循环数据流

  • 每一步是独立的应用程序
  • 数据必须完整传输

    • 待解决的问题被分为若干个子问题，每个子问题由一个独立的知识源加以理解
    • 知识源包含独立的领域知识
    • 知识源执行计算后会跟新黑板里的数据状态
    • 多个知识源 之间只能通过黑板交换知识

控制器

  • 这种更新又导致其他知识源参与计算并更新黑板，直到找到问题解为止

5. 事件系统体系结构风格

事件

  • 能够激活对象功能的动作，当发生这种动作后将给所涉及对象发送一个消息，对象便可执行相应的功能。

事件系统

主要特点 - 隐式调用

  • 事件的触发者并不知道那些构件会被这些事件影响，相互保持独立
  • 不能假定构建的处理顺序，甚至不知道那些过程被调用
  • 各个构件之间彼此无连接关系，各自独立存在，通过对事件的发布和注册实现关联

基本结构和工作原理

  • 事件源发布时间，事件管理器通知时间给事件处理器处理
  • 特点
    • 分离的交互：时间发布者并不会意识到事件订阅者的存在
    • 一对多通信：采用发布/订阅消息传递，一个特定时间可以影响多个订阅者
    • 基于事件的触发器：由事件出发过程调用
    • 异步：支持异步操作

基本构件

  • 事件源 ：发布事件
  • 事件处理器：向特定的事件进行注册
  • 事件管理器

连接机制

  • 一个事件也可能会触发其他事件，形成事件连

应用实例

调试器中的断点机制

  • 基本构件
    • 事件源：调试器
    • 事件处理器：编辑器与变量监视器
    • 事件管理器：集成开发环境
  • 处理过程
    • 编辑器与变量监视器向调试器注册，接受断点事件
    • 遇到断电，调试器发布事件，从而触发编辑器与变量检测器
    • 编辑器将源代码滚动到断点处
    • 变量检测器则跟新当前变量值并显示出来

调度机制

无独立调度模块的事件系统

  • 观察者/被观察者
  • 每一个模块都允许其他模块向自己所能发送的某些消息表明兴趣
  • 当某一模块发出某一事件时，他自动将这些事件发布给那些曾经向自己注册过此事件的模块

有独立调度模块的事件系统

  • 事件派遣模块
    • 负责接收到来的事件并派遣他们到其他模块
  • 派遣方式
    • 全广播式
      • 派遣模块将时间广播到所有的模块，但只有感兴趣的模块才取出事件并触发自身的行为
    • 选择广播式
      • 派遣模块将时间送到那些已经注册的模块中
选择广播式的两种策略：基于事件被执行的方式
点对点模式：基于消息队列（一对一）
  • 其他应用向消息队列发布事件，队列管理器会在消费者将他注册过的事件取回后删除事件。
发布订阅模式（一对多）
  • 其他应用向消息队列发布事件，队列管理器会在消费者将他注册过的事件取回搁浅到事件过期后再将事件删除

ATAM

1. 概述

  • ATAM是一种帮助涉众提出正确问题以发现潜在问题架构决策的方法

2. 目标

  • ATAM的目的不是提供精确的分析，其目的是发现架构决策所产生的风险

3. 参与人员

  • 评估小组：
    • 每个评估团队由一名领导和至少三名其他团队成员组成。

  • 项目决策者
    • 能够控制程序开发过程和授权变更的人员
    • 通常包含项目经理,为开发买单的用户,和体系结构开发人员
  • 架构涉众
    • 涉众是与要开发的系统相关的一切人和事
    • 涉众不等于用户

KWIC案例

需求获取

  • 功能性：输入 -> 循环位移 -> 排序输出
  • 非功能性：可修改，系统复用性

1. 主程序 - 子程序风格

主要思想

  • 对系统进行功能分解，是最自然的想法,也是“面向过程的编程” 的主要思路
  • 分为四个基本功能：输入、移位、排序、输出
  • 主程序按次序调用这四个模块
  • 通过共享的数据存储和无约束的读-写协议在模块之间进行数据交换

优缺点

  • 优点
    • 模块之间的数据共享
    • 不同的计算功能被隔离在不同的模块中
  • 缺点
    • 对数据存储格式的变化将会影响几乎所有的模块
    • 对处理流程的改变与系统功能的增强也很难适应，依赖于控制模块内部的调用次序
    • 这种分解也难以支持有效的复用

2. 面向对象风格

主要思想

  • 数据不再被构件直接共享，而是被封装在了Object中
  • 每个对象提供了一个接口，允许其他对象通过该接口调用对该对象内封装的数据的操作
  • 主要对象
    • Input object：负责从输入文件中读取数据并将其存储在LineStorage对象
    • LineStorage object：存储和处理字符、单词、行
    • CircularShifter object：负责对LineStorage对象中存储的数据进行循环移位
    • Alphabetizer object： 负责对循环移位后得到的数据进行排序
    • Output object：负责打印输出排序后的数据)
    • Master control object ：负责控制其他各对象中方法的调用次序

优缺点

  • 优点
    • 某一构件的算法与数据结构的修改不会影响其他构件
    • 构件之间依赖性降低，提高了复用度
  • 缺点
    • 不是特别适合功能的扩展
    • 为了增加新功能，要么修改已有的模块，要么就加入新的模块

3. 管道过滤器风格

主要思想

  • 四个过滤器：输入、移位、排序、输出
    • “输入”过滤器：从数据源读取输入文件，解析格式，将行写入输出管道
    • “循环移位”过滤器
    • “排序”过滤器
    • “输出”过滤器
  • 每个过滤器处理数据，然后将结果送至下一个过滤器
  • 控制机制是分布式的：只要有数据传入，过滤器即开始工作
  • 过滤器之间的数据共享被严格限制在管道传输

优缺点

  • 优点
    • 过程流非常直观
    • 支持复用
    • 容易修改
    • 过滤器的功能相互隔离
    • 新功能容易加入
  • 缺点
    • 无法支持交互式系统，局限性较大
    • 空间复杂性高
